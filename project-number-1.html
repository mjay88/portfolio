Learning HTML5

Unlearning the past...

Since I am learning HTML5 in the year of our lord 2021, I have'nt even been shown the proper way to use a div element yet, but I'm sure we'll get there soon. So far, it has been impressed upon me to use(from what I understand are relatively newer)sectioning elements. This practice is understood as using HTML elements semantically.

In language, semantics refers to how words are interpreted or the meaning of words. Typically, when writing something-let's say something like an editorial-we want to be as precise and concise as possible, with the goal of getting our argument across as clearly as we can. Basically we want it to make as much sense as possible to whoever is reading it, so it is best practice to avoid pedantic words and phrases or anything else that is going to distract from getting the point across, such as unconventional formatting or overly stylized formating, and use proper syntax and punctuation, so the reader is focused on your argument, and not your spelling or punctuation errors. When writing in HTML, semantic elements can be understood in the same way (in HTML, semantic elements is typically referring to HTML tags). Just like a persuasive editorial, we would want our HTML to be as precise and concise as possible, i.e.; only using div and span tags when we have an explicit reason to do so. This is not only important for someone else who may be looking at your HTML file directly, but also for web browsers which will be reading your HTML as well. Using appropriate semantic elements helps with the accessibility of our webpage. Assistive technologies such as screen readers rely on the proper use of semantic elements for those with disabilities to be able to access them. Using semantic elements helps web browsers make sense of your content, with the ultimate goal of more people seeing your page more often. A properly formatted HTML file with a good use of semantic elements will be much easier for another programmer or a co-worker to understand, so they can focus on an actual task instead of having to decipher confusing or poorly written code. We should always write our HTML while considering semantic elements, with the end goal of being as precise and descriptive as possible with our HTML tags.

Commonly used/new? elements

Element Description
section Used for grouping together thematically-related content. Sounds like a div element, but its not. The div has no semantic meaning. Before replacing all your div’s with section elements, always ask yourself, “Is all of the content related?”
aside Used for tangentially related content. Just because some content appears to the left or right of the main content isn’t enough reason to use the aside element. Ask yourself if the content within the aside can be removed without reducing the meaning of the main content. Pullquotes are an example of tangentially related content.
header There is a crucial difference between the header element and the general accepted usage of header (or masthead). There’s usually only one header or ‘masthead’ in a page. In HTML5 you can have as many as you want. The spec defines it as “a group of introductory or navigational aids”. You can use a header in any section on your site. In fact, you probably should use a header within most of your sections. The spec describes the section element as “a thematic grouping of content, typically with a heading.”
nav Intended for major navigation information. A group of links grouped together isn’t enough reason to use the nav element. Site-wide navigation, on the other hand belongs in a nav element.
footer Sounds like its a description of the position, but its not. Footer elements contain information about it’s containing element: who wrote it, copyright, links to related content, etc. Whereas we usually have one footer for an entire document, HTML5 allows us to also have footer within sections.
article The article element is another specialized kind of section. Use it for self-contained related content. The article element is useful for blog posts, news stories, comments, reviews, and forum posts. The HTML5 specification goes further than that. It also declares that the article element should be used for self-containded widgest: stock tickers, calculators, clocks, weather widgets, and the like. The WHATWG says:

A section forms part of something else. An article is its own thing. But how does one know which is which: Mostly the real anser is "it depends on author intent" http://bkaprt.com/html52e/04-05

You can have multiple article elements wihtin a section; you can have multiple section elements within an article; you cna nest sections within sections and articles within articles. It's up to you to decide which element is the most semantically appropriate in any given situation

Using HTML5 semantic elements in real life

The new elements of HTML5 give us the ability to structure our websites with intuitive divisions. Most modern conventional sites consisit of a section with a header followed by nested articles, with non essential content in an aside element, concluded with a footer. While keeping in mind the benefits of using good practices with new sectioning elements and semantic elements, these new elements were not created to replace div elements. Proper use of div elements provides our site with dynamism that is otherwise hard to accomplish.
